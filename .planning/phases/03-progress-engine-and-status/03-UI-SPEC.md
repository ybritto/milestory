## Phase Surface Contract

- Keep Phase 3 on the existing goal detail route. Do not create a separate progress screen, tab set, or dashboard preview.
- The goal detail screen must render in this vertical order:
  1. Status hero
  2. Plan-versus-actual comparison strip
  3. Progress history section
  4. Checkpoint section with progress-aware annotations
- The status hero is the dominant card on first paint. It must answer, in order: `What is my current status?`, `Why?`, and `What can I do next?`
- The progress-entry trigger must live inside the status hero for active goals and remain visible without scrolling past the fold on desktop.
- Progress history must stay on the same page and must be visible without switching tabs, accordions, or secondary navigation.
- The checkpoint section stays read-only in Phase 3, but each checkpoint card must carry enough context to relate the plan to the current cumulative state.
- Archived goals remain readable but must not expose an active progress-entry trigger or editable progress affordances.

### Layout Contract

- Desktop: use a single-column reading flow capped around `1120px`, with the hero card spanning full content width and comparison/history/checkpoints stacking beneath it.
- Large desktop only: the comparison strip may become a two-column internal layout inside its card, but the page itself remains one primary column to keep status reading linear.
- Mobile: preserve one-column flow; the hero, history, and checkpoint cards stack with clear section separation and no side-by-side metric tiles.
- Progress history entries should read like a chronological log, not a data table. Use stacked cards or rows with strong left alignment and enough white space for note text.

### Surface Style

- Keep the warm translucent card treatment from Phase 2, but make the status hero slightly stronger than the supporting cards by using a denser surface gradient and slightly larger internal padding.
- Status hero background should feel luminous and intentional, not gamified. Use subtle warm-to-surface gradients with soft blur and shadow, never high-saturation dashboard chrome.
- Supporting cards for history and checkpoints should remain lighter and quieter than the status hero so status retains visual priority.
- Use rounded surfaces consistently:
  - Major cards: `28px`
  - Secondary inset blocks: `18px` to `24px`
  - Pills and action buttons: fully rounded

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Status dot offsets, icon gaps, compact inline metadata |
| sm | 8px | Pill gaps, helper-copy spacing, stacked value-label pairs |
| md | 16px | Default card internals, comparison rows, history item spacing |
| lg | 24px | Section padding, action groups, card-to-card rhythm inside a feature block |
| xl | 32px | Major layout gaps between hero, comparison, history, and checkpoints |
| 2xl | 48px | Hero card internal breathing room and page-level section breaks |
| 3xl | 64px | Top and bottom page breathing room on desktop |

Exceptions: `44px` minimum touch target for every tappable control; `72px` top padding remains allowed on large screens; `56px` minimum height for the primary `Add progress update` button and overlay submit button.

### Spacing Rules

- The status hero must use `48px` internal padding on desktop and `24px` on small screens.
- The space between the status label, summary sentence, and comparison strip inside the hero must never collapse below `16px`.
- Progress history entries should use `16px` vertical padding for compact logs and `24px` only when a note is present.
- Checkpoint cards should use `24px` padding, with annotation content separated from base checkpoint facts by at least `16px`.
- Overlay forms should keep the input stack at `16px` gaps and separate the action row from fields with `24px`.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 24px | 600 | 1.2 |
| Display | 40px | 600 | 1.1 |

### Typography Rules

- Use exactly two weights in Phase 3: `400` and `600`.
- `Fraunces` is reserved for:
  - goal title in the hero
  - the large pace headline when status is rendered as a display moment
  - no more than one supporting emphasis line inside the hero
- `Manrope` is required for status labels, comparison values, history rows, checkpoint metadata, buttons, and form controls.
- Keep status reading tight and direct:
  - pace label pill uses `14px / 600`
  - summary sentence uses `16px / 400`
  - `Actual so far` and `Expected by today` value labels use `14px / 600`
  - numeric values in the comparison strip use heading scale, not display scale
- Do not introduce a fifth text size for progress history. Notes, empty states, and correction explanations must stay at body or label size.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F6F1E7` | Page background, overall canvas, low-emphasis surfaces |
| Secondary (30%) | `#E6DDCF` | Card fills, overlay body, comparison strip, checkpoint surfaces |
| Accent (10%) | `#1F6B5C` | Primary add-progress actions, ahead/on-pace emphasis, focus rings, active checkpoint annotations, success state highlight |
| Destructive | `#B64735` | Behind-state warning accents, archive action, blocking errors, destructive confirmation only |

Accent reserved for: primary `Add progress update` button, `On pace` and `Ahead` status pill/background treatment, focus outline, success toast accent bar, comparison delta emphasis when progress is at or above plan, and the active or current checkpoint annotation marker. Never use accent for all text, every border, or all icons across the page.

### Color Rules

- Preserve the warm neutral Milestory base from Phase 1. Do not introduce cool grays, black dashboard panels, or neon success colors.
- `Ahead` should feel energized by using the accent more boldly:
  - stronger accent pill fill tint
  - subtle warm-to-accent glow behind the hero summary zone
  - celebratory copy emphasis in ink or accent
- `On pace` should feel calm and steady:
  - soft accent tint
  - no glow stronger than the ahead state
  - avoid looking like a warning or completion state
- `Behind` should stay honest but not punishing:
  - use destructive only for the badge, rule, or key delta
  - keep the surrounding card warm and readable, not red-washed
  - keep body copy in ink/muted tones so the screen still feels usable
- Correction entries should not reuse destructive red as the primary surface color. Treat them as informative with a muted warm highlight plus a small `Correction` chip.
- Primary text remains near-ink `#1F1A17`; secondary/support text remains muted `#5F554C`.
- All state treatments must pass WCAG AA against their local surface.

## Interaction Contract

- Load goal detail and derived pace data immediately on route entry. The first meaningful hero state must render within the existing goal detail screen, not as a blank page.
- The status hero must support these explicit data states:
  - `loading`: skeleton blocks for the pace label, headline, summary sentence, and comparison strip; never a spinner-only hero
  - `ready-with-history`: pace state, summary, comparison strip, visible history list, checkpoint annotations
  - `ready-empty`: no progress entries yet; keep the goal detail page intact and show an encouraging empty-state message plus the primary add-progress action
  - `error`: page-level read failure with a retry action and a concise explanation
- The primary CTA label for active goals is `Add progress update`.
- The progress-entry overlay uses one responsive pattern:
  - desktop and tablet: centered modal with a dimmed backdrop
  - mobile: bottom sheet or drawer rising from the bottom edge
- Opening the overlay must not navigate away from goal detail. Preserve scroll position beneath it.
- Overlay fields, in order:
  1. `Date`
  2. `Progress so far`
  3. `Note` (optional)
- Overlay submit label: `Save progress update`.
- Overlay secondary action label: `Close without saving`.
- On save success:
  - close the overlay
  - refresh the goal detail payload
  - return focus to the `Add progress update` trigger
  - show an inline success message near the hero actions or as a lightweight toast
- Success message copy:
  - normal entry: `Progress updated. Your status has been refreshed.`
  - correction entry: `Correction saved. Milestory updated your status from the new total.`
- Correction detection remains backend-owned. The UI only renders the returned entry type and explanation.
- History list ordering is newest first.
- Each history row must show:
  - entry date
  - cumulative value and unit
  - optional note
  - correction indicator when entry type is `CORRECTION`
- Checkpoint annotations must clarify relation to today without implying direct editing. Use labels such as `Expected by now`, `Latest checkpoint passed`, or `Upcoming checkpoint`.
- For archived goals:
  - hide or disable `Add progress update`
  - keep history visible
  - show one read-only note explaining that archived goals no longer accept updates
- Keyboard and focus requirements:
  - opening the overlay moves focus to the dialog title or first field
  - tab order stays trapped within the overlay until close
  - `Escape` closes the overlay
  - closing returns focus to the trigger
- Motion should stay minimal:
  - one soft fade/translate for hero refresh
  - one slide/fade for overlay open and close
  - one short highlight pulse on the newly refreshed status block after save
  - no looping celebratory motion

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Add progress update |
| Empty state heading | No progress logged yet |
| Empty state body | Add your first update to see how this goal is tracking against the plan for today. |
| Error state | Milestory could not load the current progress status for this goal. Retry the page, and if the issue persists confirm the backend is running and the goal still exists. |
| Destructive confirmation | Archive goal: Archive this goal? Its plan and progress history will stay visible, but you will not be able to add new updates while it is archived. |

### Tone Rules

- Keep the product voice calm, warm, and intentional.
- `Ahead` copy may sound energized and proud, but it must stay specific to the plan comparison.
- `Behind` copy must acknowledge the gap directly and pair it with a next-step framing.
- `On pace` copy should feel steady and reassuring rather than celebratory or flat.
- Avoid analytics jargon such as `variance`, `delta model`, `projection engine`, or `trajectory`.
- Prefer direct phrases such as `actual so far`, `expected by today`, `ahead of plan`, `behind today`, `correction saved`, and `next checkpoint`.

### Status Summary Copy Contract

- `Ahead` summary pattern:
  - `You're ahead of the pace you planned for today.`
- `On pace` summary pattern:
  - `You're right where this goal expected you to be today.`
- `Behind` summary pattern:
  - `You're behind the pace you planned for today, but the year still has room to recover.`
- Comparison helper pattern:
  - `Actual so far: {value} {unit}`
  - `Expected by today: {value} {unit}`
- Correction helper pattern:
  - `Logged as a correction to reflect a lower total than an earlier update.`
